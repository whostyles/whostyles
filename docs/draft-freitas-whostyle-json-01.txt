                 The Whostyle JSON Schema Specification
                       draft-freitas-whostyle-json-01

Abstract

   This document defines Whostyle JSON, a structured, data-driven 
   format for syndicating personal typographic and cosmetic styles 
   across decentralized web platforms (e.g., the IndieWeb, Fediverse).
   By constraining user-defined styles to a strict subset of design 
   tokens and establishing an asynchronous cache lifecycle, this 
   specification enables host platforms to render third-party 
   syndicated content (such as webmentions or comments) while 
   guaranteeing layout integrity, accessibility, performance, and 
   mitigation of Cascading Style Sheets (CSS) injection vulnerabilities.

1.  Introduction

   Decentralized web interactions, such as Webmentions [W3C.REC-webmention],
   often strip the original author's visual identity when displaying 
   syndicated responses on a host platform. Prior attempts to preserve 
   visual identity ("Whostyles") relied on raw CSS injection or IFrames, 
   introducing severe security vectors (e.g., persistent Cross-Site 
   Scripting, visual deception, layout hijacking) and high computational 
   overhead.

   This specification solves these design flaws by abstracting cosmetic 
   styles into a rigid JSON [RFC8259] schema. This format translates 
   styling into non-executable design data, eliminating structural 
   layout properties (such as position, display, and float) and third-
   party asset loading (such as external web fonts).

   The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",

   BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all

2.  Schema Definition

   A Whostyle JSON document MUST be a valid JSON object containing a top-
   level key named "whostyle". The object structure consists of explicit 
   cosmetic properties defined below. Any key present in the document 
   that is not explicitly defined in this specification MUST be ignored by 
   the parsing implementation.

2.1.  The "version" Property

   The "version" property is REQUIRED and MUST be a string specifying the 
   version of this specification. For this document, the value MUST be "1.0".

2.2.  The "typography" Property

   The "typography" property is REQUIRED and specifies the font stack to 
   be applied to the container text. To prevent external asset-loading 
   exploits and arbitrary tracking, remote font URLs (e.g., @font-face) 
   MUST NOT be allowed.

   The value MUST be a string restricted to one of the following generic 
   and system-safe font-family keywords:

   * "sans-serif"
   * "serif"
   * "monospace"
   * "cursive"
   * "fantasy"
   * "system-ui"

2.3.  The "text_transform" Property

   The "text_transform" property is OPTIONAL and specifies the 
   capitalization effect of the text. The value MUST be a string 
   restricted to one of the following explicit values:

   * "none"
   * "capitalize"
   * "uppercase"
   * "lowercase"

2.4.  The "text_align" Property

   The "text_align" property is OPTIONAL and specifies the alignment of the 
   text within the container. The value MUST be a string restricted to one 
   of the following explicit values:

   * "left"
   * "right"
   * "center"
   * "justify"

2.5.  The "list_style_type" Property

   The "list_style_type" property is OPTIONAL and dictates the marker of 
   any internal list elements (<ul>, <ol>). The value MUST be a string 
   restricted to one of the following explicit values:

   * "disc"
   * "circle"
   * "square"
   * "decimal"
   * "lower-roman"

2.6.  The "border_style" Property

   The "border_style" property is OPTIONAL and specifies the line style of 
   the component container border. The value MUST be a string restricted to 
   one of the following values:

   * "none"
   * "hidden"
   * "dotted"
   * "dashed"
   * "solid"
   * "double"
   * "groove"
   * "ridge"
   * "inset"
   * "outset"

2.7.  Properties with Strict Numeric Limits

   To ensure host layout containment, numeric parameters MUST be integers or 
   floats within strict bounds. Host implementations MUST enforce these 
   boundaries by applying a clamping function (i.e., min/max validation). 
   Units MUST NOT be included in the JSON values; the host implementation 
   MUST append the pixel ("px") unit during CSS generation.

   +----------------+---------+-----------+-----------+----------------+
   | Property Name  | Type    | Min Value | Max Value | Enforced Unit  |
   +----------------+---------+-----------+-----------+----------------+
   | border_width   | Integer | 0         | 4         | px             |
   | border_radius  | Integer | 0         | 16        | px             |
   | shadow_offset  | Integer | -4        | 4         | px             |
   | shadow_blur    | Integer | 0         | 8         | px             |
   | letter_spacing | Float   | -0.5      | 2.0       | px             |
   +----------------+---------+-----------+-----------+----------------+

   * "shadow_offset": Maps directly to both X and Y offsets of the 
      "text-shadow" and "box-shadow" properties simultaneously to simplify 
      implementation constraints.

2.8.  The "theme" Object

   The "theme" property is REQUIRED and contains objects defining colors for 
   different viewing modes. It MUST contain at least one of the following sub-
   objects: "light", "dark". If the host environment does not support theme 
   switching, it MUST fallback gracefully to the alternative parameters.

2.8.1.  Color Format Constraints

   All color values within the theme objects MUST be strings representing a 
   valid hexadecimal color code. Implementations MUST validate these strings 
   against the following regular expression pattern:

   ^#([A-Fa-f0-9]{6}|[A-Fa-f0-9]{3})$

   Implementations MUST NOT execute alpha-channel transparency parameters 
   (8-digit hex codes) to prevent malicious overlapping and text concealment.

2.8.2.  Theme Property Structure

   Each theme sub-object ("light" and "dark") MUST support the following 
   keys:

   * "background": REQUIRED. The hexadecimal background color of the block.
   * "text": REQUIRED. The hexadecimal primary text color.
   * "border_color": OPTIONAL. The hexadecimal border color. If omitted, it 
      SHOULD default to the value specified in "text".
   * "link_color": OPTIONAL. The hexadecimal color for anchoring tags (<a>).
   * "link_hover_color": OPTIONAL. The hexadecimal color for anchoring tags 
      during hover states (a:hover).

3.  Processing and Host Enforcement Rules

3.1.  Strict Structural Omission

   Host implementations MUST NOT parse, translate, or inject structural layout 
   properties from the syndicated entity. Properties including, but not 
   limited to, "position", "display", "float", "clear", "width", "height", 
   "z-index", "margin", and "padding" are explicitly forbidden to ensure the 
   host platform retains absolute authority over document geometry. Remote 
   background images via "background-image" are explicitly forbidden.

3.2.  Accessibility and Contrast Validation

   Prior to rendering, the host implementation SHOULD calculate the relative 
   luminance contrast ratio between the parsed "text" color and the 
   "background" color within each theme object, in accordance with Web Content 
   Accessibility Guidelines (WCAG) 2.1 specifications.

   If the computed contrast ratio is less than 4.5:1, the host engine MUST 
   override the "text" or "background" property to its own default system 
   values to ensure legibility.

4.  Caching and Lifecycle (Lazy Polling)

   To mitigate distributed denial-of-service vectors and reduce remote network 
   dependency, host platforms MUST implement an asynchronous lifecycle for 
   third-party Whostyle JSON documents.

4.1.  The 30-Day Time-To-Live (TTL)

   When a Whostyle JSON document is successfully retrieved, the host MUST 
   cache the processed tokens locally. The local cache duration MUST be 
   limited to a maximum Time-To-Live (TTL) of 30 days.

4.2.  Lazy Polling and Revalidation

   Upon rendering syndicated content, the host MUST query its local cache:

   1. If the cached log is younger than 30 days, the host MUST render the 
      cached style instantly without external network requests.
   2. If the cached log exceeds 30 days, it is considered stale. The host MUST 
      still render the stale style immediately to ensure zero layout blocking 
      for the visitor, but it MUST dispatch an asynchronous background worker 
      (Lazy Polling) to fetch, validate, and update the remote JSON document.

4.3.  Event-Driven Invalidation (Webmentions)

   If an author updates their local Whostyle document and requires immediate 
   propagation across federated hosts before the 30-day TTL expiration, the 
   author's platform SHOULD dispatch a new Webmention update target payload 
   to the host. Upon receiving a valid update mention, the host MUST bypass 
   the TTL and trigger an immediate style revalidation.

5.  JSON Schema Validation Document

   The following JSON Schema dialect validates a Whostyle document structure:

   {
     "$schema": "https://json-schema.org/draft/2020-12/schema",
     "title": "Whostyle",
     "type": "object",
     "required": ["whostyle"],
     "properties": {
       "whostyle": {
         "type": "object",
         "required": ["version", "typography", "theme"],
         "properties": {
           "version": { "type": "string", "const": "1.0" },
           "typography": {
             "type": "string",
             "enum": ["sans-serif", "serif", "monospace", "cursive", 
                      "fantasy", "system-ui"]
           },
           "text_transform": {
             "type": "string",
             "enum": ["none", "capitalize", "uppercase", "lowercase"]
           },
           "text_align": {
             "type": "string",
             "enum": ["left", "right", "center", "justify"]
           },
           "list_style_type": {
             "type": "string",
             "enum": ["disc", "circle", "square", "decimal", "lower-roman"]
           },
           "border_style": {
             "type": "string",
             "enum": ["none", "hidden", "dotted", "dashed", "solid", 
                      "double", "groove", "ridge", "inset", "outset"]
           },
           "border_width": { "type": "integer", "minimum": 0, "maximum": 4 },
           "border_radius": { "type": "integer", "minimum": 0, "maximum": 16 },
           "shadow_offset": { "type": "integer", "minimum": -4, "maximum": 4 },
           "shadow_blur": { "type": "integer", "minimum": 0, "maximum": 8 },
           "letter_spacing": { "type": "number", "minimum": -0.5, "maximum": 2.0 },
           "theme": {
             "type": "object",
             "anyOf": [
               { "required": ["light"] },
               { "required": ["dark"] }
             ],
             "properties": {
               "light": { "$ref": "#/$defs/themeBlock" },
               "dark": { "$ref": "#/$defs/themeBlock" }
             }
           }
         }
       }
     },
     "$defs": {
       "themeBlock": {
         "type": "object",
         "required": ["background", "text"],
         "properties": {
           "background": { "type": "string", "pattern": "^#([A-Fa-f0-9]{6}|[A-Fa-f0-9]{3})$" },
           "text": { "type": "string", "pattern": "^#([A-Fa-f0-9]{6}|[A-Fa-f0-9]{3})$" },
           "border_color": { "type": "string", "pattern": "^#([A-Fa-f0-9]{6}|[A-Fa-f0-9]{3})$" },
           "link_color": { "type": "string", "pattern": "^#([A-Fa-f0-9]{6}|[A-Fa-f0-9]{3})$" },
           "link_hover_color": { "type": "string", "pattern": "^#([A-Fa-f0-9]{6}|[A-Fa-f0-9]{3})$" }
         }
       }
     }
   }

6.  Security Considerations

   Whostyle JSON addresses major security concerns associated with arbitrary 
   CSS ingestion. By avoiding raw stylesheet parsing, parsing libraries are 
   immune to CSS-based data exfiltration, blind injection attacks, and regular 
   expression Denial of Service (ReDoS) vulnerabilities. The constraint on 
   numeric parameters mitigates UI-redressing and clickjacking vectors, 
   ensuring syndicated components remain structurally constrained to their 
   allocated DOM subtrees.
